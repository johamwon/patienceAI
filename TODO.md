读完 PRD、README 和实施计划，我对项目有了完整理解。先简单复述确认我没理解偏：

这是一个面向重症/罕见病/疑难杂症患者及家属的循证医学检索 + 通俗化解释 Agent，核心卖点是"找得到、看得懂、能核验、敢上线"。技术差异化压在五位一体多智能体通俗化引擎和四层幻觉防御 + 三级风险路由上，检索层外包给 KnowS AI。合规上严守"不诊断、不开方、不替代医生"。

下面是我从患者视角和产品视角给的差异化建议。我会刻意区分"PRD 已覆盖但值得加强的"和"目前缺失、最能拉开差距的"。

先说一个判断：你现在的差异化压错了重心
PRD 把差异化叙事重心放在"五位一体 Simplification Loop"。这是很漂亮的技术差异化，但对患者来说,五个 Agent 协同他们完全无感——他们只关心三件事:我能不能信、我看不看得懂、我接下来该干嘛。

市面上的医疗 Agent(丁香医生、百度健康、各种 GPT 套壳问诊)同质化在哪?它们都在做"通用问诊/答疑",而且都在假装确定——给一个看起来权威的答案。你真正的护城河不是"五个 Agent",而是一个反直觉的定位:

做那个"诚实地告诉你不确定性、并把你武装好去见医生"的 Agent。

下面几条建议都是围绕这个重心展开的。

患者视角:四个最能打动人的差异点
1. "带着问题去看医生"——就诊准备包(最高优先级,我认为这是杀手锏)

重症患者的真实痛点不是"查不到信息",而是见医生时只有 5-10 分钟,紧张到不知道问什么,回家又后悔漏问了。

设计一个功能:患者查询后,系统不仅给解释,还自动生成一份可打印/可截图的"就诊提问清单":

针对你的情况,该问医生的 5 个关键问题(基于检索到的证据)
该主动告知医生的信息点
该索取的检查/化验项
该确认的治疗方案选项
这个功能合规风险极低(它把决策权还给医生)、患者价值极高(把焦虑转化为行动力)、而且几乎没有竞品在做。它还巧妙地把 AI 定位成"患者-医生"之间的桥梁,而非取代医生——这正好契合你的合规叙事。

2. 情绪感知,但绝不给假希望

刚确诊的李女士、照护父亲的王女士,她们打字时是带着恐惧的。大多数医疗 Agent 是冷冰冰的检索框。你已经有 Mascot 组件了,可以更进一步:

检测查询中的情绪信号(恐慌、绝望、急症倾向),调整表达的温度——但核心红线是永远不用乐观语气掩盖坏证据。伦理三原则里"不要把不确定性藏起来"应该延伸成产品的情感设计准则。
区分"信息需求"和"情绪需求"。问"我是不是快死了"时,先给的不该是生存率数字,而是引导+权威求助渠道。
这件事的难点是分寸——做不好就变成虚伪的安慰。建议把它当成表达层的约束,而不是单独堆砌"安慰话术"。

3. 证据冲突解释器——"为什么不同研究说法不一样"

患者最崩溃的体验:一篇说有效,一篇说没用,预印本和正式期刊打架。现有竞品要么只给一个答案(掩盖冲突),要么把一堆链接砸给患者(信息过载)。

你可以做一个显式的冲突调和:当检索结果存在矛盾时,主动告诉患者"目前研究结论不一致,原因可能是 A 研究样本是早期患者、B 研究是晚期",并标注证据保鲜期(这是 2021 年的共识 / 这只是会议摘要还没正式发表 / 这个试验只有 30 人)。

这直接利用了你已有的 GRADE 分级和来源标注,但把它翻译成患者能理解的"为什么",而不是冷冰冰的"低确定性"标签。

4. 研究雷达——把一次性查询变成长期陪伴

重症/罕见病是一场持续数月数年的旅程(画像 B 的张先生、画像 D 的陈先生都是)。PRD 把"个性化订阅"放在 P2,我认为应该提前到核心叙事——这是从"工具"升级到"伴侣"的关键,也是留存和差异化的根本。

让患者订阅"SMA 基因疗法"或"BTK 抑制剂耐药新方案",当有新的高质量证据(新指南、新试验、新 RCT)出现时主动推送通俗化解读。没有哪个通用医疗 Agent 在为单个患者长期追踪一种病的科研动态。

产品视角:定位、取舍与几个要避的坑
1. 把"诚实"做成品牌,把合规做成壁垒

你的合规约束(不诊断、不开方、显式标注不确定性)在竞品眼里是"束手束脚",但你应该反过来把它做成品牌人格:这个产品因为只说有证据的话、敢说"我不知道"、敢说"去问医生",反而最值得信任。在一个充斥着 AI 一本正经胡说八道的领域,"诚实"是最稀缺的差异化。

2. 窄切口,打深井

不要一上来覆盖所有疑难杂症。建议 MVP 聚焦 1-2 个高发癌种或一类罕见病(比如肺癌 + 一种你能拿到医学顾问支持的罕见病),把术语库、类比库、就诊问题模板、证据保鲜规则全部做深。深度才是护城河——通用化反而会让你和竞品一样浅。实施计划里 L2 数据集"覆盖 5 个高发癌种"我觉得对 MVP 太宽了。

3. 一个必须正视的技术坑:FKGL 不适用于中文

PRD 和实施计划反复用 FKGL/SMOG/ARI 作为可读性门控,但这三个指标都是为英文音节设计的,对中文几乎无意义。你的目标用户是中文患者,却用英文可读性指标做质量门——这会让"说得懂"这个核心指标从根上失真。需要换成中文可读性方案(如基于词频分级、句长、字频的中文可读性公式,或参考中文分级阅读语料)。这点建议尽早在需求/设计里纠偏。

4. 重新评估五位一体引擎的投入产出

五个 Agent + 编辑距离收敛 + 三套可读性指标,在 6 周 MVP 里是很重的投入,而且串行调用 5 个 Agent 会显著拉高时延(和你 p95≤3s 的目标冲突)。我的建议:把它当成可演示的技术亮点保留,但患者价值的大头(就诊准备包、冲突解释、研究雷达)不要因为它挤占资源。如果时间紧,优先保证患者侧体验,简化引擎内部轮数。

5. 患者经验 vs 循证证据的边界

患者其实很想知道"别人用了这个药什么感受"。这是巨大需求,但和循证证据是两回事。如果你要碰,必须视觉上、标签上严格区分"研究证据"和"患者自述经验",绝不混为一谈。这块要么明确不做,要么非常小心地做。

如果这些方向你认可,我建议挑其中最想落地的一个差异化能力(我个人最看好"就诊准备包"或"研究雷达")正式立项,走需求→设计→任务的 spec 流程把它细化成可实施的方案。
